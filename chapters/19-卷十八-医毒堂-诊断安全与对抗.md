# 卷十八 医毒堂：诊断、安全与对抗

系统长期运行，必有伤损；外界有意，亦会投毒。故龙场OS以“医毒堂”收束安全与运维之学，使之不再只是末尾附录，而成为整套系统得以久行的护法。医毒堂之中，又分药王谷与五毒堂两翼：前者重诊断与修复，后者重威胁研究与对抗处置。

药王谷所对治者，主要是内伤与慢病：状态漂移、记忆损坏、规则失配、错误累积、模型疲劳、协作链松动。它关注的是系统“为何越来越不像自己”。五毒堂所对治者，则是外敌与注入：memory poisoning、goal hijack、prompt injection、权限诱导、长时操控、工具链污染。它关注的是外部如何借系统之口、之手、之记忆而行其私。二者不可混同。若把外部投毒全说成心贼，便失去威胁模型的锋利；若把内部慢病全当外部攻击，便看不见自身治理之疏。

然而，分两翼，不等于分两家。真正成熟的安全治理，必须让诊、防、治、留案相互勾连。一次 memory poisoning 成功，不只要清洗受污染记忆、封锁注入口，还要回看卷八的分流为何失守、卷九的晋升为何失门、卷十的工具为何越权、卷十一的省察为何未告警、卷十六的知止为何未起效。反过来，一次内部漂移若长期不治，也会为外部攻击留下更多可乘之隙。

所以，医毒堂至少要留下三类制度。第一，威胁建模与分诊制度，明确当前龙场下真正的攻击面是什么。第二，修复与留案制度，确保每次中毒或伤损之后都有清理、回滚、责任归属与案例归档。第三，免疫反哺制度，把见过的毒理转回规则更新、训练样本、工具门禁与红队场景中，使系统越历险越不易重伤。

到此，全书的安全观也就清楚了：龙场OS不满足于“挡住坏东西”，它更要知道自己何处会伤、何处会漏、何处该养、何处该封。会防者未必会久行；会诊、会治、会留案、会反哺者，方能久行。
