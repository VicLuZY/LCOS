# 后记：如何继续扩写本书

### 小节一母稿意识：生长而非封顶

此稿非定本，乃可生长之母稿。扩写须沿“整体架构 → 子系统机制 → 失控模式 → 案例演绎 → 实施落点”推进。勿徒增字数；须增结构密度、证据密度、案例密度、机制密度。每扩一章，须自问：是否更可独立发表？是否更可回扣总图？

### 小节二优先扩写之卷

宜先厚总纲与枢纽：卷三、四、五、九、十三、十八。此数卷为后文所系，骨厚则全书立。

### 小节三图谱与附录

后续应补：龙场OS 总架构图、记忆流向图、多 agent 门派编排图、安全控制面图、知行环时序图、经验入谱生命周期图。并立术语附录，逐条给出阳明术语、系统映射、工程含义、易误读点、关联章节。

### 小节四案例卷与出版包

可增案例卷：企业研究系统、长时报告写作、工程咨询工作流、多 agent 蒸馏、组织知识传承、对抗演练。每章另制出版包：摘要、关键词、综述、主张、机制、案例、结论、附图、参考文献，使书可分可合。

### 小节五与仓库规则之自守

凡后续扩写，在本书匹配路径上，须守 `non_negotiable_style` 与 **GZAOSEAL** 所代表之古早硬约束：声部古雅而机理清晰，不以口语偷换清楚。母稿与章节若分路维护，须 deliberate 再同步，免裂为两书。

### 小节六同步脚本与工程习惯

今而后，宜以脚本将 `chapters/` 诸篇回灌母稿正文，使 YAML 元数据与章句并行不悖；每改一章，即跑构建与字数统计，勿凭目测。习惯成，则书可长而不乱，卷可增而不散，龙场OS 乃真成可演进之操作系统文本，而非一次性稿本。
