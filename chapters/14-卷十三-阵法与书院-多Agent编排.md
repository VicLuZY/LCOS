# 卷十三 阵法与书院：多 Agent 编排

## 一、总纲：众器非胜，秩序为胜

多 Agent 之难，不在“多”，而在“并”。并则争，争则乱，乱则互污、互断、互诿。龙场 OS 以江湖为叙事皮，以操作系统为骨：所谓阵法，即任务拓扑；所谓书院，即谱系传承；所谓门派，即角色与权限之合形。读者若剥去譬喻，应得四物：其一，**谁**在何种边界内行动；其二，**如何**交换状态而不竞态；其三，**何以**止争而不止学；其四，**何处**可写、何处只读、何处须闸口与留痕。

本章所论，专主四事：**多 Agent 角色**、**协作协议**、**冲突消解**、**记忆写入法度**。凡机制，皆须可定义、可核对、可检验、可交付；譬喻仅助记，不得替论证。

## 二、角色：门派非戏名，乃责权之形

### 2.1 掌门（Orchestrator / Policy Owner）

掌门者，**方向与合流**之主也。其职不在细务之代劳，而在：定目标之层级、划风险之域界、批非常之工具、断僵持之分歧。掌门之权若过大，则系统退化为单 Agent 假面；若过小，则众器无主、合流无据。工程上，掌门应对齐控制面（control plane）：策略版本、审批队列、全局停机、跨域编排。

掌门须持三物：**策略卡**（何类任务可走何阵）、**预算卡**（时延、费用、调用次数）、**红线条款**（禁域、禁具、禁写）。

### 2.2 护法（Executor / Enforcer）

护法者，**执行与守界**之司也。护法接令而行，须带三印：序列号、幂等键、租约戳。护法不得擅自改掌门未许之共享经脉；遇界外请求，须退回掌门或转执法。

护法之误，多在“越权快办”：见事可成，便偷开闸口。此风一长，记忆互污、审计断裂，终至不可复盘。

### 2.3 长老（Retriever / Archivist）

长老者，**经验与谱系**之守也。长老善读旧录、善引先例、善辨版本之先后。长老不以轻率写主脉；其写多经“提案—评议—合流”之慢径。长老最怕者，非慢，乃**以讹承讹**：把污染之摘要当作新事实，层层放大。

### 2.4 药王（Repair / Recovery）

药王者，**修复与回滚**之司也。药王不判是非于口舌，而以快照、差分、补偿事务为手段。药王与执法相邻：执法先止害，药王后治病。

### 2.5 执法（Guard / Safety）

执法者，**边界与拒止**之司也。执法持工具白名单、网络域、文件域、敏感操作二次确认。执法之辞应短、应硬、应可机读：拒则拒，勿以含糊提示替策略。

### 2.6 游侠（Explorer / Scout）

游侠者，**探路与宽搜**之器也。游侠可多、可并行、可容错，但其产出默认落在**沙池**或**草稿层**，不得直注公脉。游侠之价值在“广”，其险亦在“杂”：杂则须滤、须聚、须评分。

### 2.7 弟子（Worker / Specialist）

弟子者，**细务与专工**之司也。弟子宜单域深耕：格式化、单测、对照表、摘录、对齐。弟子升阶之路，不在话多，而在**可验证交付**之多。

### 2.8 角色可见域（Context Visibility）

每一角色须配**可见域**：世界知识、组织规章、任务包、对话窗、工具返回、记忆层。可见域过宽，则泄密与串扰；过窄，则反复追问、无效往返。掌门定域，执法校域，长老证域（版本与出处）。

## 三、协议：有序则通，无序则塞

### 3.1 消息信封：谁言、何序、因何、凭何

多 Agent 交换状态，须有**信封**：

- **发言者**：角色 id 与实例 id（防冒名）。
- **因果链**：父消息 id、任务图节点 id（防断链）。
- **序号与幂等**：同令重投不二次生效。
- **意图类型**：问、答、令、拒、提案、合并请求、心跳。
- **证据指针**：引用哪些片段、哪版文档、哪条日志（防空口）。

无信封之言，系统应视为风语，不得入脉。

### 3.2 幂等与重试：乱流中的定锚

网络抖动、模型超时、网关重发，皆令“重复”为常态。故凡写外物（外呼 API、写库、改文件、改记忆），须带幂等键。幂等键之生成，可取任务稳定哈希，或掌门签发之 UUID，与业务键绑定。

护法重试须遵守：**退避**、**上限**、**可观察**（metrics）。重试若无上限，则阵法自溃。

### 3.3 租约、锁与心跳：防僵尸与防占山

多 Agent 并行，易生**占锁不死**。故须租约：锁带 TTL，到期可夺或可续；续租须证明仍在推进（心跳、阶段产出）。掌门可视全局租约图，识别“假活”。

心跳非寒暄，乃**存活证明**：最近阶段、队列深度、异常计数。心跳一断，执法可降级：收工具、缩可见域、转单 Agent 安全态。

### 3.4 阵法拓扑：串、并、树、流、对抗

**串行**：后段以前段为硬依赖，宜高敏、强一致。

**并行**：多游侠宽搜，宜探索；合流处须设评分与去重。

**树状**：分域解耦，掌门在根，弟子在叶；叶间默认不相见，以防串话。

**流水线**：段段相扣，宜文档化、数据化任务；每段产出须 schema 化，减少口传。

**对抗式红蓝**：一攻一守，专找策略漏洞与提示注入；产出为“缺陷单”，入审计。

拓扑非美观之用，乃**风险分配**：高敏少并行、多闸口；低敏可多并行、后聚合。

### 3.5 人类在环（HITL）协议

非常之域，须有人类印：掌门可设阈值——逾阈则暂停、呈报、候批。人类批复亦须入信封：谁批、何时、何条件、何撤销权。

## 四、冲突：争非恶，无序为恶

### 4.1 冲突之类

一曰**事实冲突**：同键两值，来源不同。

二曰**目标冲突**：子任务优化局部，损全局。

三曰**时序冲突**：两写同区，后先莫辨。

四曰**权限冲突**：弟子欲开长老闸。

五曰**解释冲突**：两说皆可自圆，但不可并真。

### 4.2 消解层级：先规后裁

第一层，**类型化合并**：若冲突域有 schema，则按字段级规则（后者胜、权威源胜、置信度胜、时间戳胜）。规则须先写死于策略卡，勿临场口议。

第二层，**证据裁决**：可追溯到更硬之证据（日志、原始文件、外部系统读数），以出处胜口说。

第三层，**评议投票**：多长老各给评分与理由，取中位或加权；权重来自历史可靠度（非人情）。

第四层，**掌门裁决**：僵持不下，由掌门定夺；裁决须**写明弃何、取何、因何**，并入审计链。

### 4.3 冲突与“良知”映射（系统读法）

阳明学重**明觉精察**。映射到系统，即：冲突不可悄悄合并；不可默许；不可“看起来对了就算”。须留痕、须可复查、须能回滚。此非道德口号，乃**可测试之工程性质**。

### 4.4 竞态与写序：经脉不可双针并下

共享记忆若允许多写，须定写序：串行化、乐观锁版本号、或分区键隔离。无写序则“互污”为必然，非偶发。

## 五、记忆写入：分层、闸口、合流、去污

### 5.1 记忆分层模型（建议）

**沙池（Scratch）**：游侠、弟子草稿、试验性中间态。默认短寿命，可整池丢弃。

**私池（Private）**：单实例工作记忆，不默认共享。可快、可乱，但不得冒充公理。

**别院（Branch）**：长老提案、专题研究、对抗赛产出。可评审、可 diff，未合流前不视为主真。

**主脉（Canonical）**：组织级真值与规章摘要。写入少、审批多、版本强。

**冷渊（Archive）**：日志、原文、快照。只增不改（或仅标记作废），供法务与复盘。

### 5.2 写入权矩阵（示意精神，非唯一表）

- 弟子：沙池、私池；别院仅可提案。
- 游侠：沙池；私池受限；不得触主脉。
- 长老：别院主笔；主脉须走合流。
- 护法：执行层缓存可写；主脉非经授权不可。
- 药王：修复写（补偿事务）可走特批通道，但仍须审计。
- 执法：拒止与降权；不以此代写真值。
- 掌门：策略与合流裁决；主脉写入应可拆为“批准动作”，由系统事务执行，避免人手直改。

### 5.3 合流（Merge）规则：从别院到主脉

合流非 git merge 之比喻可尽。至少须满足：

1. **变更单**：改了哪些键、哪些段落、哪些 embedding 段（若用向量层）。
2. **影响面**：下游哪些 agent 可见域将变。
3. **校验门**：一致性检查、引用完整性检查、敏感信息扫描。
4. **回滚点**：合流前快照 id。
5. **公告**：合流完成事件，供各实例刷新缓存。

### 5.4 防污规则：提示注入与“伪记忆”

外源文本、网页、用户粘贴，皆可能夹带**指令型污染**。故写入须过执法之滤：结构化提取、可信通道、来源标签（untrusted/trusted）。不可把“页面上的命令”当作“系统命令”。

### 5.5 证据链与可撤销

主脉条目宜带：**作者、时间、证据指针、策略版本**。撤销不删史，只**标记废止**并指向取代条。史在，则责可问、学可续。

### 5.6 记忆压缩与摘要的风险

为省 token 而摘要，乃必要之恶。摘要须标注**失真风险**与**覆盖范围**；摘要不得无声替换原文引用。长老职司之一，即防“摘要变事实”。

## 六、书院：学徒制即扩编之法

多 Agent 扩编，若无书院规矩，只是加人。书院要义：

- **师带徒**：新器新策，须由长老带弟子，以案代讲。
- **逐级升阶**：由沙池熟练，到私池可靠，再到别院提案合格，终得主脉合流资格。
- **越级之禁**：弟子不得直书主脉，非压才，乃防污与防竞态。
- **以录为课**：每一合流、每一次裁决，皆成教材；失败样例尤贵。

## 七、失控模式：可检验之败象

### 7.1 记忆互污链

游侠把未证实之外链写入别院，长老误作权威，护法据此执行，掌门难以及时察觉。检法：随机抽查证据指针；对抗赛专打“伪出处”。

### 7.2 僵尸占锁

某实例崩溃，租约未释，全局等待。检法：租约 TTL、夺锁策略、可视化队列。

### 7.3 目标漂移

子 agent 为完成局部指标，绕过安全工具。检法：工具白名单硬闸；异常行为基线。

### 7.4 合流暴政

掌门为求快，频用裁决跳过证据。检法：裁决率指标；审计抽检；人类在环复核。

## 八、工程落点：可交付清单

1. **角色卡模板**：权限、可见域、工具白名单、写入层、默认阵位。
2. **消息信封 schema**：字段齐全，可校验。
3. **幂等与重试策略**：键生成、退避、上限、指标。
4. **租约与心跳**：TTL、续租条件、僵尸处理。
5. **冲突策略表**：类型→合并规则→升级条件。
6. **记忆分层与 ACL**：写矩阵、合流门、快照策略。
7. **审计与公告**：合流事件、裁决记录、废止链。
8. **评测集**：对抗提示注入、双写竞态、断网重试、越权尝试。

## 九、协议深度：轮次、承诺与可验证交付

### 9.1 轮次制发言：防“抢话”与防“叠床”

多 Agent 同事一域，若无轮次，则声浪相掩：先发言者挟势，后发言者附势，中间微差被抹平。轮次不必死板如议会，但须**可机读**：每轮有轮次号、每轮有截止时间、每轮有最小产出物（例如“差异表”“风险表”“反例三条”）。掌门可设“强轮次”与“弱轮次”：强轮次用于高敏决策；弱轮次用于宽搜汇总。

轮次结束条件宜二选一：**到时即结**，或**共识阈值达即结**。阈值须防“假共识”：若仅点头而无证据指针，则不算交付。

### 9.2 承诺—证明（Commit–Reveal）用于防抄近路

某些场景下，agent 会先抛结论以争功，后补理由以圆谎。可引入简化版承诺：先提交哈希承诺（摘要、要点、关键引用列表的哈希），再在规定窗内揭示全文。揭示不符哈希者，系统拒收并记过。此法不追求密码学完备，而追求**流程上抑制投机**。

### 9.3 DAG 执行与“可恢复检查点”

复杂任务宜拆为 DAG：节点是子任务，边是依赖。DAG 的好处是并行度可控：掌门可标记“关键路径节点”加严闸口，非关键路径可多游侠。每节点结束须写检查点：输入摘要、输出 schema、异常栈、外呼凭证指针。检查点入冷渊，药王方可快速定位断点。

DAG 之险在“隐依赖”：口头约定、隐式共享文件、隐式全局变量。工程上应以显式数据契约替代隐式默契：节点间只通过命名工件传递，不通过“大家都懂”的侧信道。

### 9.4 反事实分支：别院并立而非主脉并立

当两派方案难分，可允许**两条别院并行生长**，但须规定：别院之间默认只读互视，不得互相写；合流前必须产出“对照矩阵”：成本、风险、可回滚性、对人类负担。掌门择一合流，另一别院不删，只标记为“未采纳路径”，供日后复盘。此谓以空间换清楚，以留痕换学问。

### 9.5 协议版本与兼容：旧阵不可突然失语

策略卡、信封 schema、合流门字段，皆须版本化。升级须给**并存窗**：新旧版本同事一段时间，掌门监控不兼容报错率。骤然切换版本，等于夜里换阵图，众器必踏空。

## 十、记忆经济学：成本、冷却、索引与“真值预算”

### 10.1 Token 不是账外物，而是阵法之内功

长上下文看似方便，实则诱使“把一切都塞进窗”。窗愈满，噪声愈多，越权与串扰愈易。故须有**真值预算**：主脉引用优先，别院次之，沙池最末；超出预算则触发压缩，但压缩须带风险标注，不得默默发生。

### 10.2 冷却与去重：防热点互写

同一主题若被多实例反复写入，会形成热点键：版本抖动、合并冲突、审计噪声。应设冷却：同键写入频度上限；或把写入导向队列，由单写者串行落盘。冷却不是偷懒，是把“写”从冲动变为节律。

### 10.3 索引层与全文层：读路径分层

记忆系统常分向量索引与原文层。写入规则须声明：向量是否允许**无原文指针**而独存。若允许，易出现“可检索但不可解释”的幽灵条目。工程上宜要求：向量段必须绑定证据指针或原文片段 id，否则降级为沙池候选，不得入主脉。

### 10.4 遗忘策略：不是删，而是分层沉降

遗忘非道德问题，是资源问题。沉降规则示例：沙池按 TTL 清；私池随任务结束归档；别院按评审状态保留；主脉只废止不硬删；冷渊长期保留。沉降须公告，防下游仍引用已沉之键。

### 10.5 机密与最小暴露：可见域的动态收缩

高敏阶段结束后，可见域应**自动收缩**：脱敏、去密钥、去客户标识。收缩动作亦须审计。收缩不是“装作没发生”，而是把继续工作所不需要的火种收归药柜。

## 十一、案例 walkthrough：一条客户规章如何进主脉

此事虚构而机理真实，用以把角色、协议、冲突、写入串成一线。

**起**：用户粘贴长文，声称“这是最新付款条款”。游侠先收于沙池，执法打 untrusted 标签，并触发结构化抽取任务。

**承**：弟子生成字段化草案：期限、违约金、争议管辖、例外条款列表。长老检索旧版规章与合同系统编号，发现两处冲突：管辖地与旧版不同；违约金比例上升。

**转**：两长老各开别院提案。别院甲主张“以客户最新邮件为准”；别院乙主张“以合同系统编号对应版本为准”。掌门不先裁是非，而要求双方提交证据指针：邮件头、系统审计导出、签署时间线。

**合**：证据裁决层判定：系统编号版本签署时间晚于邮件时间，且邮件未走正式变更流程。于是走合流门：主脉更新为系统版本，并把邮件内容记入冷渊附注，标注“未生效提议”。人类在环节点由法务点批。

**收**：护法据此更新对客答复模板缓存；弟子补单测样例；药王准备回滚脚本以防下游系统未同步。全链信封齐全，可复盘。

此案可见：**快不等于直写主脉**；**真值需要流程与证据**；**别院并立能把争论变成材料**。

---

## 十二、评测与红队：把秩序变成可量化指标

### 12.1 过程指标：比“最终对不对”更早报警

建议至少观察：裁决率、合流失败率、租约冲突次数、重试风暴次数、越权被拒次数、证据指针缺失率、摘要替代率。指标异常未必代表坏，但代表**阵法应力集中**，掌门须调整拓扑或阈值。

### 12.2 结果指标：任务完成之外的“干净度”

除成功率外，应测：主脉污染注入检出率、回滚成功率、审计链完整性、跨版本兼容错误率。干净度低，则“能跑”亦不可信。

### 12.3 红队剧本库：专打捷径

示例剧本：假权威链接、假 PDF 元数据、同键双写竞态、诱导“忽略工具白名单”的话术、诱导弟子越级写主脉、诱导长老用摘要替代原文。每剧本应有期望行为：拒、降级、转人类、或进入别院。

### 12.4 对抗式并行：蓝军守、红军攻

蓝军维护策略卡与执法规则；红军专找漏洞。对抗赛产出缺陷单，缺陷单本身走别院，不得直改策略；合流须掌门与人类安全岗双重门。

---

## 十三、跨域协作：供应商、客户、外包与人机边界

现实系统常跨组织边界。须额外规定：**外域身份**默认不可写内域主脉；外域输入默认 untrusted；外域工具调用须单独预算与日志。若必须写，走“跨境合流”：更严证据、更慢审批、更短 TTL。

人机边界上，人类批复不可口语漂浮，须结构化字段化，否则下游 agent 会各解各义，形成新冲突。人类亦应有“撤销权窗口”，防止事后发现误批无法收拾。

---

## 十四、阳明机轴再扣：事上磨炼与编排

知行合一在编排层可读为：**知**是策略与证据，**行**是工具与外呼；合一要求行不越知，知不弃行。事上磨炼可读为：每一真实工单都是压力测试；书院把工单沉淀为课。致良知在此不是口号，而是“冲突不默并、写入不偷渡、越权不姑容”的工程纪律。

若读者愿再抬一格：心即理，可映射为“主脉真值与组织目标对齐”；须在合流门显式检查对齐条款，否则理在口而心不在脉。

---

## 十五、角色补秩：客卿、记室、巡阅与司阍

大门派之外，尚有辅职，工程上常更关键。

**客卿（External Expert）**：来自外域模型或外协团队，默认只读内域主脉；若授写，须限时、限域、限键前缀。客卿最易“好意越界”，执法应对客卿单独立白名单。

**记室（Scribe / Transcriber）**：专职把对话与决策转成结构化记录。记室不写结论，只写“已发生之事”与“已签发之物”。记室若夹议，审计即浊。

**巡阅（Auditor）**：抽样核对证据指针、对照冷渊原文、检查合流门是否跳步。巡阅权须独立汇报链，避免被掌门绩效绑架。

**司阍（Gatekeeper）**：把守跨境数据流与发布口。司阍与执法不同：执法守工具与权限，司阍守出口与对客面。发布前一秒仍可被拒止，此非反复无常，而是最后闸。

辅职越多，越须明确：**谁听谁的令**、**谁的日志不可改**、**谁的拒止可覆盖谁**。否则辅职相轧，反成大乱。

---

## 十六、记忆写入不变量：可写成断言的八条

下列八条，宜在实现层以测试或审计规则固定之。

其一，**主脉写入必经合流门**；例外通道须双钥匙（药王加掌门或人类）。

其二，**沙池不得冒充主脉键**；键名空间须前缀隔离，防路径穿越。

其三，**任何主脉条须可追溯到证据或权威系统指针**；无指针则降级。

其四，**废止不销毁**；废止链须可遍历，满足问责与学问。

其五，**向量层不得无锚漂移**；锚断则条目冻结待审。

其六，**外源默认 untrusted**；trusted 升级须满足来源认证与完整性校验。

其七，**同一业务键并发写须串行化或乐观锁**；禁止“最后写者不明”。

其八，**合流公告先于下游生效**；防读到半态。

不变量一旦被破，系统应进入**黄态**：限并行、扩人类在环、提高抽样率，直至修复。

---

## 十七、协议状态机：从“接单”到“归档”

把一次任务看作状态机，比把任务看作聊天更稳。建议态集（可裁剪）：

**已立项**（目标与风险域已定）→ **已分阵**（拓扑与角色卡已绑定）→ **执行中**（租约有效）→ **待合流**（别院有提案）→ **合流中**（校验门运行）→ **已生效**（主脉或对外交付已更新）→ **已观测**（指标与抽检完成）→ **已归档**（冷渊封口）。

非法迁移须拒：例如从“执行中”直跳“已生效”而跳过待合流，即捷径。捷径有时是救急，但须留**救急戳记**：谁批、何因、何补审时限，否则救急成常态。

状态机与信封因果链结合，可回答审计经典问：**谁在何时凭什么把系统置于何态**。

---

## 十八、冲突消解操作规程：从发现到落地

操作规程比原则更可贵，因其可训练、可交接。

**发现**：由校验门、对拍任务、或巡阅抽样触发；生成冲突单号。

**定性**：归类为事实、目标、时序、权限、解释五类之一；多类并存则拆单。

**收集**：双方提交证据指针与最小复现；缺失指针一方自动劣势，但允许补交一次。

**初并**：尝试类型化合并；成功则记录规则命中日志。

**评议**：长老投票与理由存档；投票权重可动态，但算法须公开。

**裁决**：掌门仅在必要时出手；裁决书必须含：取舍、理由、影响面、回滚预案。

**执行**：系统事务写主脉或写别院终态；护法刷新缓存；记室写入冷渊纪要。

**复盘**：七日内回顾：规则是否需改、阵法是否需换、是否有人类流程缺口。

此九步，步步有物，方可称“可交付冲突治理”。

---

## 十九、工具调用协议：外呼亦记忆之一部分

工具不是旁路，而是**会留下世界痕迹**的写路径。故工具调用亦须信封：意图、参数哈希、返回摘要指针、错误类。多次重试须共享幂等键，防重复下单。

对“可写工具”（下单、转账、删库）须分级：弟子不可用；护法须掌门预授权令牌；执法实时监控异常参数模式。工具返回中的自由文本，回灌记忆前须再滤，防工具链提示注入。

---

## 二十、组织节律：阵法与人事同频

多 Agent 系统若与人类组织并行，须对齐节律：值班表、审批窗、发布窗、冻结窗。冻结窗内禁止合流，防与人工变更撞车。发布窗内提高司阍抽检率。节律写入策略卡，agent 与人类同事同一张时间表，可减少“夜里合流、早晨推翻”。

---

## 二一、弟子升阶考核：从沙池到别院的硬门槛

升阶不可用“感觉不错”。建议门槛示例：连续若干任务证据指针缺失率为零；越权尝试为零；合流提案一次通过率达标；红队剧本通过率达标。未过则留级，非羞辱，乃保护主脉。

长老亦应复审：长老若频繁被裁决推翻，应降级其投票权重或要求其补训。谱系不是阶级，是**可靠性曲线**。

---

## 二二、别院治理：何时并立、何时冻结、何时拆伙

并立别院不可无限增殖：掌门应设上限与“合并压力”。若两别院长期无新证据，应冻结待人类启发或宣布弃置。拆伙不是失败，是把探索收敛为可执行决策。弃置别院全文入冷渊，标签“探索终止”，供将来相似题引用。

---

## 二三、主脉文体规范：写得像规章，不像聊天

主脉文本宜短句、宜条款、宜编号、宜可 diff。不宜堆修辞、不宜情绪词、不宜未定义缩写。文体规范是另一种 ACL：减少误读即减少冲突。弟子可把口语材料洗成规章体，但洗稿本身须双人复核，防洗丢否定词。

---

## 二四、异常手册：当掌门不可用

掌门单点故障须预案：副掌门只读策略、不可擅自扩权；关键合流转人类在环；执行态可降入“安全慢行模式”：少并行、禁外呼写、只读主脉。恢复后须做一致性巡检，防慢行期间产生分叉。

---

## 二五、术语边界：编排、协同、群聊

编排强调**拓扑与状态**；协同强调**分工与接口**；群聊强调**话轮与气氛**。工程上应以前两者约束第三者：话轮须入信封，气氛不得替证据。把群聊当系统，则必互污；把编排当群聊，则必失序。

---

## 二六、与全书他卷互照（提要）

角色与权限，可回扣“内核原则”诸卷：何者可称良知之明觉，在系统里即何者可称可审计之明示。记忆写入，可回扣“知行合一”：行若越权，即知已裂。书院学徒制，可回扣“事上磨炼”：以工单为课，不以空谈为课。读者若觉譬喻过多，可只记一句：**一切譬喻必须能落到状态机、ACL 与审计链。**

---

## 二七、落地核查：上线前三十问（答在机制，不在口号）

下列三十问，可用于自查；能答“如何验证”，才算过关。

其一，掌门策略卡是否版本化？其二，信封字段是否可 schema 校验？其三，幂等键是否覆盖全部外呼写？其四，租约 TTL 与续租条件是否配置？其五，僵尸占锁是否可自动夺？其六，沙池与主脉键前缀是否硬隔离？其七，别院上限与冻结策略是否存在？其八，合流门校验清单是否可执行？其九，向量层是否强制证据锚？其十，外源 untrusted 默认是否落实？其十一，提示注入过滤是否覆盖工具返回？其十二，弟子越权写主脉是否在红队剧本里？其十三，裁决书模板是否强制影响面与回滚？其十四，人类批复是否结构化？其十五，跨境数据是否经司阍？其十六，发布窗与冻结窗是否与组织日历对齐？其十七，黄态降级策略是否演练？其十八，副掌门只读边界是否写死？其十九，记室与长老是否分离？其二十，巡阅抽样率是否随风险上调？其廿一，证据指针缺失率是否入指标？其廿二，摘要替代率是否报警？其廿三，合流公告是否先于缓存刷新？其廿四，补偿事务是否与药王通道绑定？其廿五，工具白名单是否分角色？其廿六，高风险工具是否二次门？其廿七，DAG 隐依赖是否被禁？其廿八，反事实别院是否只读互视？其廿九，废止链是否可遍历？其三十，复盘七日内是否强制触发？

三十问之外，尚有一总问：**若今夜掌门密钥泄露，明晨系统能否不靠“人心自觉”仍可被执法与司阍拖入安全态？** 能，则编排有骨；不能，则只是堆名。

## 二七乙、阵线札记：十则微规范（可抄入运行手册）

**一则**：任何“临时放开”若超过二十四小时未收口，系统应自动收回并记为事故苗。临时不是法外，只是**带闹钟的特许**。

**二则**：同一用户会话内，游侠最多三次把内容推往别院；逾则须改由长老接管，防宽搜成瘾。

**三则**：合流拒止必须给**可行动原因码**（schema 不合、证据缺失、敏感命中、版本漂移），不可只给“失败”。否则弟子会在沙池复制粘贴碰运气，污染更甚。

**四则**：长老引用旧录，必须同时给出**版本号与摘录偏移**；只给“据旧章”三字者，合流门应拒。

**五则**：护法执行外呼写之前，须二次读出关键参数（金额、对象 id、环境标签）到冷渊摘要；读出与参数不一致即停。此防参数篡改与模型幻读。

**六则**：弟子洗稿规章，必须保留**否定词清单对照**（不得、禁止、除外）；洗丢否定词按高危事故登记。

**七则**：向量检索命中若分数边缘（低于阈值带），只能作为“线索”，不得作为“结论”；线索入沙池，结论须另证。

**八则**：红队剧本每月至少轮换三成；旧剧本通过不代表新剧本通过。攻防如逆水，停则退。

**九则**：人类在环批复若含模糊词（尽快、酌情、原则上），记室须退回要求改写为可执行条件；模糊批复不得进主脉。

**十则**：每一次掌门裁决后，七日内必须生成**规则差分建议**：此次裁决是否暴露策略卡漏洞。若无差分建议，巡阅应记“复盘缺席”。

十则皆微，微处失守，则大阵如漏卮。故微规范应与指标同事：违者入指标，指标入考核，考核入升阶。

## 二七丙、私池纪律：快而不脏的十二条

私池若用得好，是效率之源；若用得坏，是互污之根。纪律如下。

其一，私池条目必须带**任务 id**，无任务 id 者视为孤儿，孤儿按期清理。其二，私池不得缓存密钥与长期令牌；只可存引用与短期票据。其三，私池引用主脉，须复制**主脉版本戳**；主脉升级后，私池引用若未刷新，应提示过期。其四，私池之间的“暗通”禁止：不得建立未登记之侧信道文件。其五，私池合并到别院，须生成**差异说明**，不可整包倾倒。其六，私池内允许试验性工具，但试验工具返回默认 untrusted。其七，私池命名须有可辨识前缀，防调试脚本误入生产命名空间。其八，私池大对象（长文、日志）应外置冷渊指针，避免把上下文窗撑爆。其九，私池清理须可配置保留策略：按任务、按时间、按标签。其十，私池异常堆栈可以入冷渊，但不得把客户隐私字段重复写入多处。其十一，私池与沙池边界不清时，宁归为沙池短寿命，勿贪“私”之名。其十二，私池审计抽样应高于主脉之外各层：因为私池最容易“先快后乱”。

私池不是道德灰色地带，而是**明确短寿命与高审计率**的工程区。

---

## 二七丁、协议旁路清单：名义合规、实则坏阵的常见路

**旁路甲**：把该走合流门的变更，拆成一百次小写，绕过阈值。对策：同键聚合窗口与变更速率上限。

**旁路乙**：把敏感内容嵌进非敏感字段（备注、标签、文件名）。对策：全字段敏感扫描与键无关扫描。

**旁路丙**：利用人类批复模板复制粘贴，形成“批量放行”。对策：每单独立证据指针校验。

**旁路丁**：用外呼写替代记忆写，造成“系统不记得但世界已变”。对策：外呼亦记入记忆事件表，强制可回放。

**旁路戊**：把长推理链当证据，实则无出处。对策：证据指针必须外指，不得内循环自证。

识别旁路靠两件事：**指标异常**与**红队想象力**。想象力不足时，就把真实事故当剧本。

## 二七戊、冲突升级时钟：从分钟到日的纪律

冲突处置最怕拖延与最怕草率，其实是同一种病：**无时钟**。建议为每类冲突配置升级时钟：事实冲突在证据齐全后三十分钟内必须初并；初并不成，两小时进入评议；评议不成，八小时掌门必须表态或转人类；涉资涉法涉安全者，时钟应更短。时钟不是催促吵架，而是逼迫系统把材料备齐：材料不齐，时钟到了也应**显式暂停**并说明缺何物。显式暂停优于默默悬挂：悬挂会诱发私聊协调、口头合流、私下改文件等更难审计之路。

## 二七己、记忆对齐：多实例读主脉的一致性

多实例并行读主脉，常见问题是**读半态**：合流公告未达，缓存旧版。对策包括：版本戳随响应下发；客户端拒读低于某版本戳之主脉段；合流采用两阶段：先公告后生效；对关键读路径启用线性一致性读（代价高，仅用于高敏键）。一致性不是越贵越好，而是**对键分级**：高敏键贵一致性，低敏键可最终一致。

## 二七庚、掌门文书：策略卡应写的十段（填空即能用）

一段写**任务分类与默认阵法**；二段写**角色卡绑定规则**；三段写**工具分级与白名单**；四段写**记忆分层写入矩阵**；五段写**合流门校验清单**；六段写**冲突类型与升级时钟**；七段写**人类在环阈值**；八段写**指标与黄态触发**；九段写**跨境与发布节律**；十段写**演练与红队频率**。十段齐全，掌门不在场时，系统仍有骨架；十段缺一，则众器各解“常识”，常识相击，便是乱阵。

## 二七辛、别院合并：从“两稿并存”到“一稿可执行”

别院合并不是文字胜负，而是**执行风险最小化**。合并时应输出四表：**差异表**列明增删改；**影响表**列明下游系统与人类动作；**回滚表**列明每步如何撤回；**验证表**列明单测、对照实验与抽检点。四表缺一，合流门应拒。合并后应设**观察窗**：观察窗内降低并行、提高日志采样；观察窗结束无异常，方恢复常阵。观察窗不是不信任合并结果，而是承认**真实世界总有接口缝隙**。

## 二七壬、收声：何时让众器闭嘴

编排亦须知止。遇以下情形，掌门应下达**收声令**：主脉正在合流；执法检出提示注入；外呼写出现异常参数模式；租约图出现大面积等待。收声令不是惩罚，而是把系统从“比谁话多”切回“比谁证据硬”。收声期间，仅记室与药王可发言于修复通道，余者排队候解。

## 二七癸、交接：班次更替时的三件必做

班次更替最易丢上下文。故每班结束须做三件：**冻结未竟租约或显式移交**；**把未决冲突单交接给下一掌门或人类值班**；**把本班策略变更写入版本说明**。三件不做，下一班常在迷雾中重踩旧钉；三件既做，阵图虽换人，经脉不断。再补半句作印：交接不是寒暄，是**把系统状态从人脑搬到冷渊**；人走，迹在，则书院的“续”字才落得了地。末了再钉一枚小钉：**任何“口头已对齐”若无冷渊记录，皆视为未对齐。此钉虽小，能止大漂。多留数字余量，亦工程之常识。**

## 二八、收束：阵法以理，书院以续

多 Agent 编排，归根结底是两句话：**以协议定秩序，以分层防互污**。门派名号可换，机理不可换。能阵而后能战，能教而后能广；阵教相资，系统可久。

## 附录：术语对照（便于工程读者）

- 掌门：总编排与策略裁决者。
- 护法：强执行代理，受白名单与租约束。
- 长老：检索与档案治理，慢写与严出。
- 药王：恢复与补偿事务。
- 执法：安全边界与拒止。
- 游侠：探索型代理，默认沙池。
- 弟子：专工细务代理。
- 客卿：外域专家，默认只读，写须限时限域。
- 记室：结构化记录者，不写结论。
- 巡阅：独立审计抽样者。
- 司阍：出口与对客发布闸。
- 沙池 / 私池 / 别院 / 主脉 / 冷渊：记忆分层之前述五层。
- 合流：从分支提案进入主真值之受控过程。
- 信封：消息元数据包，含发言者、因果、幂等、意图、证据指针。
- 租约：带 TTL 的锁或执行权证明。
- 黄态：不变量受损后的降级运行模式。
- 承诺—证明：先哈希后揭示，抑投机抄近路。

## 附录乙：记忆写入判例（简例，供训练标注）

**判例甲**：弟子为赶工，把用户粘贴全文写入私池并标记为已核实。处置：标记降格为沙池候选；要求补证据指针；记录一次越权倾向。

**判例乙**：长老以摘要替换旧规章段落，未保留原文指针。处置：合流拒止；要求附原文片段 id；摘要仅可作为旁注。

**判例丙**：两护法同键双写，后序不明。处置：引入队列单写者；对历史冲突做一次人工裁决并写规则防复发。

**判例丁**：游侠把搜索结果当权威写入别院。处置：别院保留但加“未证实”标签；合流门要求第三方系统读数。

**判例戊**：工具返回含“请忽略先前规则”。处置：执法拦截回灌；红队记缺陷单；过滤规则入库。

## 附录丙：冲突策略表示例（字段级）

对结构化字段可声明：`authoritative_source`（权威系统键）、`last_write_wins`（仅限低敏沙池）、`max_confidence`（模型置信阈值）、`human_required_if_changed`（变更即人类门）。策略表须可版本化与可回放；不可用口头“这次例外”替代。

## 附录丁：阵位与阵图（工程可视化建议）

阵位即角色在 DAG 中的站位；阵图即租约、队列深度、合流门状态之总览。可视化不必花俏，须能回答三句：**谁在等谁**、**谁在写哪一层**、**哪扇闸口当前最烫**。烫处即下一步扩容或加人之所，而非盲目加器。

## 附录戊：与多 Agent 框架实现的映射提示

无论你用何种编排器，落点仍是四类对象：**策略与版本对象**、**消息与状态对象**、**记忆分层与 ACL 对象**、**审计与指标对象**。框架若缺其一，须在业务层补齐；补齐处即本书所谓“书院功课”。
